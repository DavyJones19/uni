--- Conceptos rápidos (antes de los archivos) ---

• Next.js App Router: carpetas bajo `app/` definen rutas. `app/page.tsx` = ruta "/".
• Client Component: archivo con "use client" — React corre en el navegador (hooks, clicks).
• Route Handler: `app/api/.../route.ts` — funciones GET/POST que corren en el SERVIDOR
  (Node). Ahí puedes usar variables de entorno secretas y llamar APIs externas.
• Arrow function (función flecha): `const suma = (a, b) => a + b` — forma corta de
  funciones, útil como callback (p. ej. `.map(x => x * 2)`).
• async/await: `async function f() { const r = await fetch(...) }` — await espera
  la Promise sin bloquear la interfaz en el cliente.
• TypeScript `type` / `interface`: describen formas de datos para el compilador.
• Paginación (en este proyecto): los alumnos llegan todos en un array desde la API;
  la tabla **parte el array en el cliente** (`slice`) y muestra "Mostrando X–Y de Z"
  y "Página N de M". No es paginación en el servidor (no hay `?page=` en la API).
• Proxy Pattern: Los Route Handlers (`app/api/`) actúan como intermediarios entre
  el navegador y las APIs externas. El cliente nunca toca directamente URLs externas;
  todo pasa por `/api/...` en tu servidor.

--- DIAGRAMA DE FLUJO DE LA APLICACIÓN ---

1. INICIO (SIN TOKEN)
   → Navegador carga app/page.tsx
   → Muestra LoginForm (components/LoginForm.tsx)
   
2. LOGIN
   Usuario ingresa credenciales
   → POST /api/login (Route Handler)
   → Proxy del servidor consulta EXTERNAL_LOGIN_URL
   → Respuesta con token → Guardado en localStorage
   → isAuthenticated = true
   
3. POST-LOGIN (CON TOKEN)
   → Página recarga: now isAuthenticated == true
   → Muestra Navbar + dashboard (componentes de catalogos)
   → Navbar incluye navegación a:
      • /catalogos/usuarios - Tabla de usuarios
      • /catalogos/grupos - Tabla de grupos
      • /catalogos/administradores - Tabla de administradores
   
4. SELECCIÓN DE GRUPO (en dashboard)
   Usuario selecciona grupo → GrupoSelector.tsx
   → useEffect carga grupos: GET /api/grupos
   → onClick "Buscar" → POST /api/alumnos { grupo: "X" }
   → Tabla de alumnos refresca con paginación cliente

--- ARCHIVO: app/api/login/route.ts (ROUTE HANDLER — PROXY DE LOGIN) ---

Propósito: Endpoint /api/login en el SERVIDOR. Proxy seguro hacia EXTERNAL_LOGIN_URL.

Líneas clave:
  1-3   Imports: NextRequest, NextResponse de next/server.
  
  5-7   Variables de entorno (solo servidor):
    - LOGIN_URL = EXTERNAL_LOGIN_URL || ""
    - isDev = NODE_ENV === "development"
    
  10-13 POST handler:
    - Valida que LOGIN_URL esté configurada (si no, 503 Service Unavailable).
    - Lee el body JSON del cliente (credenciales).
    
  15-25 fetch() hacia la API externa:
    - method: "POST"
    - body: JSON de las credenciales del usuario.
    - signal: AbortSignal.timeout(30000) — máx 30 segundos, no cuelgues.
    
  27-48 Manejo de respuesta:
    - Si !response.ok (4xx/5xx): retorna error + status original.
    - Si ok: parsea JSON y retorna al cliente.
    - Si la API devuelve texto plano: lo envuelve en { data: text }.

    IMPORTANTE: El cliente NUNCA ve EXTERNAL_LOGIN_URL. Todo pasa por este
    servidor intermedio, así el token y URLs secretas permanecen en el servidor.

--- ARCHIVO: app/api/grupos/route.ts (ROUTE HANDLER — LISTADO DE GRUPOS) ---

Propósito: GET /api/grupos — proxy que obtiene la lista de grupos desde API externa.

Líneas clave:
  1-2   Comentario: Route Handler en servidor, sin "use client".
  
  5-8   Variables de entorno:
    - GRUPOS_URL, EXTERNAL_API_TOKEN, EXTERNAL_API_TOKEN_TYPE ("Bearer").
    
  10-15 buildAuthHeader(request):
    - Prioridad 1: Authorization del cliente (token del login del usuario).
    - Prioridad 2: Token fijo en .env (EXTERNAL_API_TOKEN) si el cliente no envía.
    
  17-22 type GrupoOption: estructura { id, label, value } para el combo.
  
  24-65 normalizeGroups(payload):
    - Normaliza respuestas de distintas APIs externas:
      * Array puro: [grupo1, grupo2] → convierte a GrupoOption.
      * { data: [...] } → extrae .data.
      * Valores numéricos o strings: cada uno es un grupo.
      * Objetos complejos: busca claves como GRUPO, grupo, nombre, Nombre, etc.
    - Deduplicación: usa un Set para no repetir valores.
    - Retorna: Array<GrupoOption>.
    
  FLEXIBILIDAD: Si tu API externa devuelve { GRUPO: "A", ID_GRUPO: 1 },
  se normaliza a { id: "1", label: "A", value: "A" } automáticamente.

--- ARCHIVO: app/api/alumnos/route.ts (ROUTE HANDLER — LISTADO DE ALUMNOS) ---

Propósito: POST /api/alumnos — obtiene filas de alumnos para un grupo específico.

Líneas clave:
  1-3   Comentario: Route Handler en servidor.
  
  5-8   Variables de entorno:
    - ALUMNOS_URL, ALUMNOS_METHOD (GET o POST, default POST).
    - Tokens de autenticación (igual que grupos).
    
  10-17 buildAuthHeader(): igual que en grupos — client auth o fallback a .env.
  
  19-26 extractRows(payload):
    - Busca el array de filas en: payload directo, payload.data, payload.rows,
      payload.resultado. Maneja distintas estructuras de respuesta.
    
  28-70 POST handler:
    - Valida ALUMNOS_URL configurada.
    - Lee { grupo } del body del cliente.
    - Construye headers con auth.
    
    SI ALUMNOS_METHOD === "GET":
      - Si URL incluye {grupo}: reemplaza placeholder → https://api.ext/alumnos/A.
      - Si no: agrega query → https://api.ext/alumnos?grupo=A.
    
    SI ALUMNOS_METHOD === "POST" (default):
      - Si URL incluye {grupo}: reemplaza placeholder.
      - Si no: envía JSON { grupo: "A" } en el body.
    
    FLEXIBILIDAD: Tu API externa puede ser GET con {grupo} en path, GET con
    ?grupo=X en query, o POST con JSON. Configura con env vars.

--- ARCHIVO: app/(dashboard)/GrupoSelector.tsx (SELECTOR CON SEARCH) ---

Propósito: Componente de selección de grupo con búsqueda interactiva y deduplicación.
Es un Client Component que mantiene UI state: búsqueda, dropdown abierto, índice
destacado para navegación con teclado.

Líneas clave:
  1     "use client"
  
  3-7   type GrupoOption: { id, label, value }
  
  9-18  type GrupoSelectorProps: props que recibe el componente.
  
  20-28 dedupeByValue(groups):
    - Quita duplicados de un array de GrupoOption filtrando por value.
    - Usa un Set<string> para rastrear valores vistos.
    
  30-35 React.forwardRef<HTMLInputElement, GrupoSelectorProps>:
    - Permite que el padre pase un `ref` y acceda al input DOM directamente.
    - Sintaxis: React.forwardRef<tipo_del_dom, tipo_props>(función).
    
  40-48 Refs internos:
    - containerRef: para detectar clicks fuera (cerrar dropdown).
    - innerInputRef: referencia al <input>, expuesta via forwardRef.
    
  50-60 useMemo(() => dedupeByValue(groups), [groups]):
    - Recalcula lista sin duplicados SOLO cuando `groups` cambia.
    - Optimización: evita recomputar si props no cambian.
    
  62-72 filteredGroups con useMemo:
    - Filtra uniqueGroups por coincidencia parcial del nombre/valor.
    - Solo recalcula si uniqueGroups o normalizedQuery cambian.
    
  74-86 useEffect: cierra dropdown si haces click fuera del componente.
  
  CARACTERÍSTICAS AVANZADAS:
    - Navegación con teclado: flechas arriba/abajo, Enter para seleccionar.
    - Búsqueda en tiempo real: filtra a medida que escribes.
    - Deduplicación automática: maneja duplicados de la API.

--- CÓMO CONFIGURAR VARIABLES DE ENTORNO (.env.local) ---

Crea un archivo `.env.local` en la raíz (NO en Git):

  # Login
  EXTERNAL_LOGIN_URL="https://tu-api-externa.com/auth/login"
  
  # Grupos (lista de grupos disponibles)
  EXTERNAL_GRUPOS_URL="https://tu-api-externa.com/grupos"
  
  # Alumnos (lista de alumnos por grupo)
  EXTERNAL_ALUMNOS_URL="https://tu-api-externa.com/alumnos"
  EXTERNAL_ALUMNOS_METHOD="POST"  # O "GET" si tu API lo prefiere
  
  # Autenticación general (si tu API externa no usa login por usuario)
  EXTERNAL_API_TOKEN="eyJhbGciOiJIUzI1NiIsInR5cCI..."
  EXTERNAL_API_TOKEN_TYPE="Bearer"  # O "Basic", "Token", etc.

Después de modificar .env.local, reinicia el servidor (Ctrl+C, pnpm dev).

--- FLUJO DE DATOS PASO A PASO (EJEMPLO COMPLETO) ---

1. Usuario ingresa "admin" / "pass123" → hace click "Entrar".
   
2. LoginForm.onSubmit → app/page.tsx.handleLogin() ejecuta:
   POST /api/login { Usuario: "admin", Pwd: "pass123" }
   
3. app/api/login/route.ts:
   - Lee el body.
   - Hace fetch a EXTERNAL_LOGIN_URL (en .env.local).
   - Respuesta: { token: "abc123def456" }.
   - Retorna al cliente: { token: "abc123def456" }.
   
4. Navegador recibe token:
   - setToken("abc123def456")
   - localStorage.setItem("auth_token", "abc123def456")
   - isAuthenticated = true → re-render.
   
5. Página muestra post-login UI con Navbar.
   Usuario navega a /catalogos/usuarios.
   
6. Página carga → useEffect envía: GET /api/grupos.
   
7. app/api/grupos/route.ts:
   - buildAuthHeader busca Authorization: "Bearer abc123def456" en la request.
   - Hace fetch a EXTERNAL_GRUPOS_URL con ese header.
   - Respuesta externa: [{ GRUPO: "A", ID_GRUPO: 1 }, { GRUPO: "B", ID_GRUPO: 2 }].
   - normalizeGroups(): convierte a [{ id: "1", label: "A", value: "A" }, ...].
   - Retorna al cliente: ese array normalizado.
   
8. Navegador recibe grupos:
   - GrupoSelector renderiza <select> con opciones A, B, etc.
   
9. Usuario selecciona "A" y hace click "Buscar".
   
10. GrupoSelector.onSearch → Tabla ejecuta:
    POST /api/alumnos { grupo: "A" }
    
11. app/api/alumnos/route.ts:
    - Extrae { grupo: "A" } del body.
    - Construye Authorization header (token del usuario).
    - Hace fetch a EXTERNAL_ALUMNOS_URL con { grupo: "A" } en el body.
    - Respuesta: { rows: [{ id: 1, nombre: "Juan" }, { id: 2, nombre: "María" }] }.
    - extractRows(): obtiene el array de filas.
    - Retorna al cliente: ese array.
    
12. Navegador recibe alumnos:
    - DataTablePriority renderiza tabla con 2 filas.
    - Footer: "Mostrando 1–2 de 2", "Página 1 de 1".

--- CAMBIOS COMUNES: HOW-TO ---

¿Cómo agregar una nueva columna a la tabla de alumnos?
→ Busca en la página /catalogos/usuarios (u donde uses TablaAlumnos).
→ En el array `columns`, agrega:
    { header: "Teléfono", accessorKey: "telefono" }
→ Asegúrate que la API externa incluya ese campo en el objeto fila.

¿Cómo cambiar el método de /api/alumnos de POST a GET?
→ En .env.local:
    EXTERNAL_ALUMNOS_METHOD="GET"
→ Reinicia el servidor.
→ app/api/alumnos/route.ts automáticamente construirá una URL con query:
    ?grupo=A en vez de JSON en el body.

¿Cómo agregar más capos a la búsqueda del GrupoSelector?
→ En GrupoSelector.tsx, la línea de filtro:
    return value.includes(normalizedQuery) || label.includes(normalizedQuery);
→ Agrega más condiciones:
    || description.includes(normalizedQuery)

¿Cómo cambiar el tamaño de página por defecto?
→ En DataTablePriority.tsx, busca `const [pageSize, setPageSize] = ...`
→ Cambia el valor inicial (ahora probablemente 10).

¿Cómo agregar un botón de logout en la Navbar?
→ En components/navbar.tsx, agrega:
    <button onClick={() => {
      localStorage.removeItem("auth_token");
      setToken("");
      window.location.href = "/";
    }}>
    Logout
    </button>

ARCHIVO: app/page.tsx  (página principal — Client Component)

Hay comentarios didácticos también en el propio código fuente.

"use client" — marca el archivo como Client Component (sin esto, no usarías

Imports: React, `LoginForm`, `GrupoSelector`, `TablaAlumnos`, y el **tipo**

`type RowData = Record<string, unknown>` — cada fila es un objeto con claves string.

  `https://mistalentos.mx/FORMATOS/Talentos_{id_usuario}.pdf` usando `id_usuario`
  de la fila (el campo puede estar oculto como columna pero sigue en `row`).

`DEFAULT_COLUMNS` — plantilla de la **tabla completa**: Grupo, nombre completo,
  Usuario, contraseña, correo, fechas, métricas, tests, columna **DESCARGAR RESUMEN**
  con `render` que muestra un botón solo si `ind_terminado` es 1 / "1" / true.

`DEFAULT_COLUMNS_VISTA_SIMPLE` — plantilla de la **segunda tabla** (demo): solo
  Grupo y Nombre completo (mismo `renderNombreCompleto`). No incluye PWD ni el
  botón de resumen: **no se "ocultan" con `HIDDEN_COLUMN_IDS`**, simplemente **no
  están en esta plantilla**.

`HIDDEN_COLUMN_IDS` — `Set` de **nombres de propiedad** de la API que no quieres
  como columna automática (ni duplicada). Ej.: `AP`, `AM`, `NOMBRE`, `id_usuario`,
  `ind_terminado`, etc. No borra datos del objeto `row`; el `render` del botón
  y `renderNombreCompleto` siguen leyendo esos campos. **Las dos tablas pueden
  compartir el mismo `Set`**: no "chocan" porque `buildColumns` se llama dos veces
  y solo **lee** el Set; cada llamada devuelve un **array de columnas distinto**.

`buildColumns(rows, defaults, hidden, incluirExtras?)`
  • A partir de `defaults` arma `fromConfig` (map con `accessorKey`, filter por
    `hidden`).
  • Si **`incluirExtras` es false** (vista simple): devuelve **solo** `fromConfig`.
    Ahí es por lo que no aparecen PWD ni columnas extra: **no están en `defaults`**
    y no se añaden extras.
  • Si **`incluirExtras` es true** (tabla completa): además añade columnas extra
    por claves de `rows[0]` que no estén en `existingIds` ni en `hidden`.

`normalizeClientGroups`
  Convierte respuestas heterogéneas (string, número, objeto) a `GrupoOption[]`.
  `??` (nullish coalescing): usa el valor de la derecha solo si la izquierda es
  null o undefined.
  `Set` evita duplicados por `value`.

`export default function Home()` — Next usa este componente en la ruta "/".

  useState — token, grupos, `rows` (alumnos del grupo), flags de carga, error, etc.

  Dos `useMemo` sobre el mismo `rows`:
    • `columns` = `buildColumns(rows, DEFAULT_COLUMNS, HIDDEN_COLUMN_IDS, true)`
    • `columnsVistaSimple` = `buildColumns(rows, DEFAULT_COLUMNS_VISTA_SIMPLE, …, false)`
  Así una sola carga de datos alimenta **dos tablas** con columnas distintas.

  `Boolean(token)` — sesión iniciada o no.

  useEffect(..., [token])
         Efecto: cuando `token` cambia, si está vacío limpia grupos; si hay token,
         define `loadGroups` (async), hace fetch a `/api/grupos` con cabecera
         `Authorization: Bearer ...`, parsea JSON y actualiza estado.
         `AbortSignal.timeout(30000)` — cancela la petición si pasa de 30 s.
         `try / catch / finally` — maneja errores y quita loading siempre.

  handleLogin — POST a `/api/login` con `{ Usuario, Pwd }`; lee varias claves
         posibles de token en la respuesta.

  useCallback `loadRows` — POST a `/api/alumnos` con Bearer; rellena `rows`.

  handleGroupChange / useRef + handleSearchClick — grupo seleccionado y búsqueda.

  JSX (en **este** archivo `page.tsx`, no solo en `TablaAlumnos.tsx`)
         Sin token → `<LoginForm />`. Con token → `<GrupoSelector />` y **dos**
         bloques `<section>`: "Tabla completa" con
         `<TablaAlumnos columns={columns} data={rows} />` y "Vista simple" con
         `<TablaAlumnos columns={columnsVistaSimple} data={rows} pagination={false} />`.
         Mismos `rows`, distintas props `columns`. Fragment `<>...</>` agrupa
         sin un `<div>` extra. Opciones de paginación: ver componente.

PERSISTENCIA DE TOKENS (localStorage)
================================================================================

Problema: Si el usuario recarga la página, pierde la sesión.
Solución: `localStorage` guarda el token en el navegador de forma persistente.

En app/page.tsx:
  1. Al hacer login exitoso:
     localStorage.setItem("auth_token", tokenStr)
  
  2. Al montar el componente (useEffect con [] dependencias vacías):
     const savedToken = localStorage.getItem("auth_token");
     if (savedToken) setToken(savedToken);
  
  3. Al hacer logout (botón en Navbar):
     localStorage.removeItem("auth_token");
     setToken("");

Así el usuario permanece logueado incluso después de cerrar/reabrir el navegador,
hasta que haga logout explícitamente.

NOTA: localStorage es accesible desde JavaScript, por lo que tokens no deben
incluir información sensible más allá del acceso a la sesión (ver sección
"Consideraciones de Seguridad").

================================================================================
CONSIDERACIONES DE SEGURIDAD
================================================================================

1. NUNCA expongas URLs de APIs externas en el cliente (HTML/JavaScript).
   → Usa Route Handlers (app/api/...) como proxy intermedio.
   → Variables de entorno EXTERNAS solo en .env.local del servidor.

2. Authorization headers SIEMPRE deben ir en Route Handlers (no en fetch del
   navegador si contienen secretos).
   → app/api/alumnos/route.ts construye el header con el token.
   → El cliente no ve el token fijo de .env.local, solo el suyo propio.

3. HTTPS en producción — localhost es OK para desarrollo, pero en Vercel/Netlify
   u otro hosting, todas las requests deben ser HTTPS para proteger tokens.

4. Tiempo de expiración del token:
   → Si tu API devuelve expiry, valida en el cliente o servidor.
   → Implementa refresh token flow si es necesario.
   → De lo contrario, el usuario permanece logueado indefinidamente
     (riesgo si deja el navegador abierto en una máquina compartida).

5. CORS:
   → Next.js por defecto acepta requests del mismo origin (localhost:3000).
   → Si tu API externa está en otro dominio, asegúrate de CORS headers en ella.
   → Si es en tu control, agrega Access-Control-Allow-Origin en el backend.

6. Validación de entrada:
   → Los Route Handlers validan que `grupo` no esté vacío (grupo.trim()).
   → Siempre limpia y valida datos en el servidor antes de usarlos.

DEBUGGING: PROBLEMAS COMUNES
================================================================================

Problema: "Error al iniciar sesión"
  → Verifica .env.local: ¿EXTERNAL_LOGIN_URL es válida?
  → Reinicia el servidor: Ctrl+C → pnpm dev
  → Abre DevTools (F12) → Network tab → busca "login" → revisa response.

Problema: Dropdown de grupos vacío o muestra "Falta configurar EXTERNAL_GRUPOS_URL"
  → Verifica .env.local: ¿EXTERNAL_GRUPOS_URL está definida?
  → ¿El token se guardó correctamente? Revisa DevTools → Application → localStorage.
  → ¿La API externa devuelve JSON válido? DevTools → Network → click en "grupos"
    → Response tab.

Problema: Tabla de alumnos no se carga
  → Mismo proceso: revisa Network tab en DevTools → POST a /api/alumnos.
  → ¿El payload { grupo: "..." } se envió?
  → ¿La API externa devolvió error o status 500?

Problema: Token se pierde al recargar la página
  → Verifica que localStorage.getItem/setItem se llamen correctamente.
  → Revisa DevTools → Application → localStorage → ¿auth_token está?
  → Si no aparece, el saveToken en login no se ejecutó (error anterior).

Problema: "Cannot read property 'data' of undefined"
  → Un campo esperado en la respuesta de la API falta.
  → Ejemplo: normalizeGroups busca `data.GRUPO` pero la API devuelve `data.nombre`.
  → Abre DevTools, copia la respuesta y ajusta las claves en el código o en la API.

Problema: Paginación no funciona (muestra todas las filas de una vez)
  → Verifica que `pagination={true}` esté en `<TablaAlumnos ... />`.
  → Revisa que `defaultPageSize` sea un número > 0.
  → Si aún así falla, busca `slice(startIndex, endIndex)` en DataTablePriority;
    asegúrate de que `currentPage` y `pageSize` se actualicen con clicks.

  {

  }

================================================================================
MEJORES PRÁCTICAS
================================================================================

1. Tipos TypeScript:
   • Define `type` o `interface` para cada respuesta de API.
   • Esto evita errores en tiempo de compilación (no runtime).

2. Manejo de errores:
   • Siempre muestra error al usuario: `error && <p className="...">` + estado.
   • En el servidor, valida y devuelve status HTTP correcto (400, 401, 500).

3. Loading states:
   • Desactiva botones durante carga: `disabled={loading}`.
   • Muestra spinner o texto "Cargando...".

4. Re-renders innecesarios:
   • Usa `useMemo` para datos costosos (columnas, filas formateadas).
   • Usa `useCallback` para funciones pasadas a componentes hijos.

5. Eventos de teclado:
   • Implementa Enter en inputs (ayuda UX): `onKeyDown={handleKeyPress}`.
   • Flechas arriba/abajo en dropdowns: GrupoSelector lo hace automático.

6. Móvil-first CSS:
   • Usa clases Tailwind responsive: `md:`, `lg:` para breakpoints.
   • Prueba en navegador (F12 → Device Toolbar) con ancho de móvil.

7. Logging en desarrollo:
   • Mantén console.log durante desarrollo; retíralos en producción.
   • o usa condicionales: `if (isDev) console.log(...)`.

8. Versionado de API:
   • Si tu API externa cambia, versiona en URLs: `/v1/alumnos` → `/v2/alumnos`.
   • Actualiza .env.local, no código hardcodeado.

CASOS DE USO AVANZADOS
================================================================================

¿Cómo implementar logout?
  → En Navbar, agregar botón con onClick:
    () => { 
      localStorage.removeItem("auth_token");
      setToken("");
      // opcional: window.location.href = "/";
    }

¿Cómo refrescar el token automáticamente?
  → Agrega un useEffect que escuche cambios de token.
  → Si token es caduca, haz refresh request a /api/refresh.
  → Actualiza token en estado y localStorage.

¿Cómo exportar datos a Excel?
  → El proyecto ya importa `xlsx` (en package.json).
  → Ejemplo:
    const workbook = XLSX.utils.book_new();
    const sheet = XLSX.utils.json_to_sheet(rows);
    XLSX.utils.book_append_sheet(workbook, sheet, "Alumnos");
    XLSX.writeFile(workbook, "alumnos.xlsx");

¿Cómo agregar paginación en el servidor?
  → POST /api/alumnos envía { grupo, page, pageSize }.
  → Route Handler construye ?page=1&pageSize=10 en la URL externa.
  → API devuelve { rows: [...], total: 100, page: 1 }.
  → DataTablePriority usa `total` para calcular "Página N de M" correcto.

¿Cómo integrar una API GraphQL?
  → En Route Handler, en vez de fetch GET/POST JSON, usa fetch de query GraphQL.
  → Construye `query { alumnos(grupo: ...) { id, nombre } }`.
  → Parsea response y devuelve al cliente igual que hoy.

ACTUALIZACIÓN: BOTONES DE PAGINACIÓN PRIMERO Y ÚLTIMO
================================================================================

Cambio realizado (19 de mayo de 2026):
Se agregaron dos botones nuevos en la paginación de DataTablePriority.tsx:

Antes:
  • Anterior (página actual - 1)
  • Siguiente (página actual + 1)

Ahora:
  • Primero (ir directamente a página 1)
  • Anterior (página actual - 1)
  • Siguiente (página actual + 1)
  • Último (ir directamente a la última página)

Implementación en app/(dashboard)/DataTablePriority.tsx (líneas aprox. 330-365):
  • Botón "Primero": onClick={() => setCurrentPage(1)}
    disabled={displayPage <= 1} → deshabilitado en la primera página
    
  • Botón "Último": onClick={() => setCurrentPage(totalPages)}
    disabled={displayPage >= totalPages} → deshabilitado en la última página

Beneficio UX:
  → Usuarios con muchas páginas pueden navegar rápidamente a inicio/final.
  → Misma línea de botones, mismos estilos (border, hover, disabled state).

Código agregado (fragmento):
  <button type="button" ... onClick={() => setCurrentPage(1)}>Primero</button>
  <button type="button" ... onClick={() => setCurrentPage((p) => Math.max(1, p - 1))}>Anterior</button>
  <button type="button" ... onClick={() => setCurrentPage((p) => Math.min(totalPages, p + 1))}>Siguiente</button>
  <button type="button" ... onClick={() => setCurrentPage(totalPages)}>Último</button>

Los botones heredan las clases de estilos de los botones anteriores/siguiente y tienen
el mismo comportamiento responsive (wrapping en móvil, row en escritorio).
